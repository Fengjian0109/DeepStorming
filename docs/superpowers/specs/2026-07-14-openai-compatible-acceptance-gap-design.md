# OpenAI-compatible 真实验收缺口补完设计

- 日期：2026-07-14
- 关联范围：D1 真实云 Provider 手动验收、D8 自用版发布候选
- 背景：DeepSeek 真实 Provider 验收已经完成，但 OpenAI-compatible 真实端点验收仍缺少可用的 `HTTPS base URL + model + API key`。当前需求不是伪造完成，而是把这块整理成可随时恢复执行的正式设计与文档口径。

## 1. 目标

在不持有真实兼容端点凭据的前提下，完成以下工作：

1. 明确 D1 的“部分完成”口径，避免把未执行的真实验收误记为完成。
2. 固化 OpenAI-compatible 真实验收的恢复条件、执行顺序、完成标准和安全边界。
3. 把当前状态、路线图和验收清单同步到规划文档，保证后续任何一次回到该主题时都能直接继续。

本设计不新增代码、不修改测试、不以 mock 或推断替代真实验收结论。

## 2. 范围与非目标

### 范围

- 更新真实云 Provider 验收总清单。
- 更新当前开发状态文档。
- 更新软件设计收敛路线图。
- 明确未来恢复 OpenAI-compatible 真实验收时的最小输入、执行路径和记录规范。

### 非目标

- 不在没有真实端点的情况下宣称 OpenAI-compatible 验收完成。
- 不新增为了“看起来完成”而存在的伪自动化。
- 不改变 Provider 领域模型、Gateway 行为或课堂主链路实现。

## 3. 方案选择

本轮比较了三种方案：

1. 最小补档方案：只补设计与规划文档，不改代码。
2. 本地模拟增强方案：增加更多 mock / e2e 覆盖，提前演练错误路径。
3. 阶段性完成方案：只在路线图里标记阻塞，不补恢复执行设计。

最终选择方案 1，并吸收方案 3 的状态口径：

- 文档中明确 D1 已完成 DeepSeek 真实验收；
- OpenAI-compatible 保持 pending；
- 阻塞原因是缺少真实兼容端点与凭据，而不是实现缺陷；
- 后续一旦补齐真实端点，直接按既有矩阵执行，不重新讨论口径。

## 4. D1 状态口径

统一使用以下表述：

- D1 当前状态：部分完成。
- 已完成范围：DeepSeek 真实 Provider 验收；D8 自用版发布候选相关基础检查已推进。
- 未完成范围：OpenAI-compatible 真实端点手动验收。
- 未完成原因：当前缺少可用的真实兼容端点与凭据，不属于本地实现缺陷。
- 风险判断：不影响当前 DeepSeek 自用闭环，但会影响“多 Provider 真实可用性”的发布信心。
- 恢复方式：补充真实 `HTTPS base URL + model + API key` 后，按 O-01 至 O-08 执行。
- 整体完成定义：OpenAI-compatible 真实验收记录补齐并通过敏感信息扫描后，D1 才能整体转为完成。

必须同时保留一条边界说明：

> 在没有真实兼容端点的前提下，现阶段只做验收设计与记录准备，不以 mock、单测或文档推断替代真实验收结论。

## 5. 恢复执行设计

未来恢复 OpenAI-compatible 真实验收时，固定按以下顺序执行：

1. 准备输入：`HTTPS base URL`、`model name`、`API key`。
2. 先做 Provider 级验证：O-01、O-02、O-03。
3. 再做课堂链路验证：O-04、O-05。
4. 最后做错误与恢复验证：O-06、O-07、O-08。
5. 完成后立即做敏感信息扫描与验收记录归档。

约束：

- 没有真实 HTTPS 端点时，不得把 OpenAI-compatible 标记为完成。
- 只完成连接测试，不算完成。
- 必须至少覆盖：连接成功、课堂生成成功、取消成功、至少一种稳定错误映射验证、重启恢复成功。
- 只记录稳定结论，不记录 Key、Authorization header、原始响应、完整 prompt 或完整用户学习内容。

## 6. 文档落点

本轮修改 3 份规划文档，职责分离如下：

- `docs/planning/provider-cloud-release-acceptance.md`
  - 作为真实 Provider 验收总清单；
  - 补充 OpenAI-compatible 真实验收恢复条件、执行顺序和完成定义。
- `docs/planning/current-status.md`
  - 只表达当前项目状态；
  - 明确 D1 处于“部分完成”，阻塞点为真实兼容端点缺失。
- `docs/planning/software-design-completion-roadmap.md`
  - 只表达路线图口径；
  - 把 D1 区分为 “DeepSeek acceptance done / OpenAI-compatible pending”。

## 7. 验证方式

本轮是文档设计补完，验证标准为：

1. 三份规划文档与本 spec 的口径一致。
2. 不出现“OpenAI-compatible 已完成真实验收”的误导表述。
3. `current-status` 中分支信息与当前工作区一致。
4. 后续阅读者只靠文档即可知道：
   - 当前缺什么；
   - 一旦补齐真实端点该怎么做；
   - 何时能把 D1 标记为真正完成。

## 8. 完成定义

本设计在以下条件满足后视为完成：

- spec 已写入仓库；
- 三份规划文档已同步；
- 文档内部不存在占位符、互相矛盾或把“待真实验收”误写为“已完成”的表述。
